= Governance – Design Prinzipien
:page-toc: right

== GP1 – Governance is Enforceable
Policies müssen technisch erzwingbar oder automatisch überprüfbar sein.

Beispiele:
* Region Pinning via Policy-as-Code
* Mandatory Tags via IaC
* IAM Guardrails via Role Policies

== GP2 – Everything is Versioned
Governance-Artefakte gehören ins Repository:

* Controls (YAML)
* Policies
* Architekturentscheidungen (ADR)
* Dependency Register
* Subprocessor Lists

Keine Excel-Shadow-IT.

== GP3 – Least Privilege by Default
Zugriff ist minimal, zeitlich begrenzt und review-bar.

Governance steuert:
* IAM
* Break-Glass
* Service Accounts
* CI/CD Rechte

== GP4 – Observable Governance
Governance ohne Logging ist Illusion.

Alle privilegierten Aktionen:
* sind geloggt
* sind review-bar
* sind alertbar

== GP5 – Sovereignty by Design
Jurisdiction, Data Residency, Key Ownership und Egress Control
werden architektonisch berücksichtigt – nicht nachträglich.

== GP6 – Exit is a Mandatory Control
Lock-in-Risiko ist ein Governance-Risiko.

Governance verlangt:
* dokumentierten Exit-Plan
* getestete Restore-Szenarien
* IaC Portabilität
